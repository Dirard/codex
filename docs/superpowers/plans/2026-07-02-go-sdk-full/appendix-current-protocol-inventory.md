# Appendix: Current Protocol And Go SDK API Mapping Seed

This appendix is a reviewed seed inventory extracted from the current stable v2 schema bundle, complete shared schema bundle, and `codex-rs/app-server-protocol/src/protocol/common.rs` at plan time. Stage 2 must populate Go-owned mapping inputs from the reviewed rows here, then generate `sdk/go/internal/protocodex/current_protocol_inventory.generated.md` from schema, Rust-owned manifest extraction, `resourceAPIMappings`, and `serverHandlerMappings`. The generated extraction is authoritative for current completeness, but public Go owner/signature/test/docs decisions must come from these reviewed mapping rows. If the manifest intentionally changes visibility or ownership, the generated inventory must include the explicit `sdkVisibility`, owner, exception reason, and review note.

## Reviewed Go Resource API Mapping Seed

The tables in this section are the reviewed input for `resourceAPIMappings`. They name expected owners, wrapper files, wrapper names, `SignatureConventionID`, concrete `CompileCallsite`, unit tests, safe integration owner or reason, docs/example owner, and review note. Stage 2 must copy these reviewed values into `resourceAPIMappings` and then generate inventory from them; Stage 5 acceptance must use the generated signature/callsite data, not method-prefix inference.

For every SDK-public row, `SignatureConventionID` is one of:

- `thin`: `Wrapper(ctx, protocol.<Payload>) (protocol.<Response>, error)` or no-param equivalent.
- `handle-start`: wrapper returns a typed handle plus the start response when the wire method starts a stream/session. If the wire payload contains connection-scoped identity owned by the SDK lifecycle row, the public root wrapper must use root package options and the SDK must generate/inject identity; generated `protocol.*Params` stay available only through `Raw()`.
- `handle-followup`: wrapper is a method on the typed handle named in `Expected root wrapper`; the handle owns/injects resource identity and the public method accepts only operation-specific data/options.
- `high-level`: wrapper uses the named high-level options/result types defined in Stage 4 or Stage 5.

Every SDK-public row below carries a concrete `CompileCallsite`; if a row exposes both handle and root entry points, both callsites are listed. `Safe integration` is either a checked integration test file or an explicit reason why CI must use mocked, OS-specific, matrix-backed, or not-applicable validation.
